#### 6.1 开发策略
- **分阶段推进**：按照"基础框架 -> 交易闭环 -> 个人中心 -> 联调验收"四阶段有序推进，降低耦合风险。
- **审查补齐机制**：阶段完成后立即审查未完成项，避免遗漏，确保每个阶段完整交付。
- **数据先行**：先建立 mock 数据层（`trade-store.js`、`user-store.js`），再开发页面逻辑，确保数据结构一致。

#### 6.2 代码组织
- **通用组件抽离**：`page-shell`、`product-card`、`state-view` 等组件在多个页面复用，减少重复代码。
- **数据层统一管理**：购物车、订单、用户、收藏等状态统一存储在 `utils/` 下，便于后续接入真实接口。
- **路由清晰**：`app.json` 中页面路径按模块组织，便于维护和查找。

#### 6.3 问题规避
- **链路闭环验证**：每个页面开发完成后，立即验证上下游跳转，避免链路断裂。
- **状态同步**：`plan.md` 和开发日志实时同步，确保进度可追踪。

- **需求**：解决小程序请求可能遇到的跨域问题（虽小程序本身不受浏览器同源策略限制，但需确保 OPTIONS 预检等场景正常）。
- **实现**：在 `WebMvcConfig.java` 中添加 `addCorsMappings()` 方法：
  - 允许所有来源（`allowedOriginPatterns("*")`）。
  - 允许常用 HTTP 方法（GET、POST、PUT、DELETE、PATCH、OPTIONS）。
  - 允许所有请求头，暴露 `Authorization` 和 `X-Trace-Id`。

- **问题**：`TRACE` 级别日志刷屏且 `SQL type unknown` 信息无实际意义。
- **调整**：将 `StatementCreatorUtils` 降为 `WARN`，仅保留 `JdbcTemplate` 的 `DEBUG` 日志。

##### 第二阶段：集成 p6spy 代理组件
- **需求**：打印带实参的完整 SQL，而非 `?` 占位符模板。
- **实现**：
  1. `pom.xml` 添加 p6spy 依赖（版本 3.9.1）。
  2. `application-dev.yml` 数据源配置调整：
     - URL 前缀改为 `jdbc:p6spy:mysql://...`。
     - 驱动类改为 `com.p6spy.engine.spy.P6SpyDriver`。
  3. 新增 `spy.properties` 配置文件：
     - 日志输出格式：`[SQL] cost=Xms | conn=X | SELECT ...`。
     - 使用 Slf4J 作为日志门面。
     - 排除调试类别的冗余日志。
  4. 调整日志级别：
     - p6spy 日志设为 `INFO`。
     - 原生 JDBC 日志降为 `WARN`，避免重复输出。

### 3. 问题与处理

#### 3.1 小程序 CORS 认知纠正
- **误区**：误认为小程序会受浏览器 CORS 限制。
- **事实**：小程序 `wx.request` 不是浏览器环境，不受同源策略约束。
- **真正限制**：需在小程序后台配置合法域名、确保网络可达性（真机调试时需使用局域网 IP）。

#### 3.2 SQL 日志输出格式问题
- **初始现象**：`JdbcTemplate` 只打印 SQL 模板（`SELECT ... WHERE id = ?`），看不到实际参数。
- **解决方案**：引入 p6spy 代理，拦截并重构 SQL 语句，将参数填入占位符。
- **最终效果**：`[SQL] cost=3ms | conn=5 | SELECT id, name FROM categories WHERE id = 1 LIMIT 1`。
