# 开发流程规则

## 新增页面/交互开发规范

在当前迭代阶段，**新增页面或交互功能**必须遵循以下流程：

### 1. 功能清单确认（必须先执行）

在开始编码前，必须列出以下信息并等待用户确认：

| 项目 | 必须说明 |
|------|----------|
| **页面路径** | 新增页面的完整文件路径 |
| **页面功能** | 该页面的核心功能描述 |
| **页面结构** | 主要模块、组件、列表项等 |
| **空态处理** | 无数据时的展示方案 |
| **交互逻辑** | 点击、跳转、操作反馈等 |

### 2. 接口评估（必须先执行）

| 评估项 | 说明 |
|--------|------|
| **已有接口** | 列出可复用的现有接口 |
| **需新增接口** | 如需后端新增接口，必须明确说明 |
| **后端功能** | 如需后端配合开发，必须提前说明 |

### 3. 等待用户确认

**在得到用户明确确认前，不得开始编码。**

确认方式：
- 用户回复"确认"
- 用户明确同意开发

### 4. 开发执行

确认后按清单执行开发，包括：
- 创建页面文件（.js, .json, .wxml, .wxss）
- 更新相关页面跳转逻辑
- 注册页面到 app.json
- 封装 API 调用（如需要）

### 5. 文档更新（必须执行）

开发完成后，必须更新以下文档：
- `update.md` - 记录新增功能、改动文件、接口信息
- `introduce.md` - 如新增接口，需记录接口说明

### 示例流程

```
用户：售后点击时跳转到售后页面

AI：
## 新增功能清单

### 1. 售后列表页
**页面路径**：miniprogram/pages/aftersale/list/index

| 项目 | 说明 |
|------|------|
| 页面标题 | 售后 |
| 页面结构 | 标题栏 + 售后列表 |
| 空态展示 | "暂无相关订单" |
| 列表项 | 售后单号、类型、状态、时间 |

### 2. 我的页入口
- 点击"售后"图标跳转售后列表

### 3. 接口评估
- 复用现有：`GET /api/v1/after-sales`
- 后端已支持，无需新增

## 待确认
1. 是否继续开发？
2. 售后列表项字段是否满足需求？

用户：确认

AI：[开始开发并更新文档]
```
